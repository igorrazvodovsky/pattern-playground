# Learnability

The capability of a system to enable users to achieve and maintain competence with minimal training demand. It is not just about initial "ease of use", but about the structural support for transition from novice to expert.

## Dimensions

### Temporal

- Initial learnability, e.g. performance of a [first-time](../?path=/docs/activities-onboarding--docs) user.
- Extended learnability, i.e. [change](../?path=/docs/activities-mastery--docs) in performance over time.

### Subject

- Interface: Can the user operate the system?
- Domain: Does the system help users understand the domain it embodies?

The tool frequently (always?) encodes domain knowledge—process logic, business rules, institutional decisions. Learning to use it well *is* domain learning. These dimensions interact: a user may be proficient with the interface but unfamiliar with certain domain areas, or understand the domain but struggle with the tool.

## Training demand

The cognitive cost required to move from initial to extended learnability. This cost differs by subject:

- Interface training demand: effort to operate the system fluently
- Domain training demand: effort to understand what the system represents

{/* Sometimes these compete (complex interface obscures domain concepts); sometimes they align (well-designed interface *teaches* domain through use). */}

## Scaffolding

Temporary support structures that help users perform tasks they couldn't yet do alone.

- For interface skills, effective scaffolding fades as competence grows—training wheels that eventually come off.
- For domain understanding, scaffolding may *deepen* rather than fade. As users master basics, they become ready for richer [explanations](../?path=/docs/actions-sensemaking-explanation--docs) of why the system works as it does.

## Cognitive levels as a design heuristic

[Bloom's taxonomy](https://en.wikipedia.org/wiki/Bloom%27s_taxonomy) provides a useful lens for two decisions that the concepts above leave underspecified:

1. *What level does the user need to reach?* Typical UI learning plateaus at Apply — users need to remember where things are, understand what they do, and operate them. Domain learning often requires Analyse or Evaluate — understanding *why* the system works as it does, or judging whether a recommendation is correct. Knowing the target level sharpens what "extended learnability" means for a given context.

2. *What kind of scaffolding to provide?* A tooltip that aids recall is different from an explanation that supports analysis. Scaffolding has not just an *intensity* dimension (how much support) but a *cognitive level* dimension (what kind of thinking it supports).

Some rough correspondences between cognitive levels and patterns in this library:

- *Remember* — [annotation](../?path=/docs/actions-sensemaking-annotation--docs), consistent layout, familiar affordances
- *Understand* — [explanation](../?path=/docs/actions-sensemaking-explanation--docs), [transparent reasoning](../?path=/docs/activities-transparent-reasoning--docs)
- *Apply* — [onboarding](../?path=/docs/activities-onboarding--docs), direct manipulation
- *Analyse* — [bot](../?path=/docs/activities-bot--docs) (contextual enrichment), comparison views
- *Evaluate* — [transparent reasoning](../?path=/docs/activities-transparent-reasoning--docs) (judging system suggestions)
- *Create* — [mastery](../?path=/docs/activities-mastery--docs)-stage tools, composition patterns

{/* ## Interaction modalities */}
{/* Different command interfaces support different skill levels. */}
{/* - **Pedagogic:** Commands that teach via inspection — menus, dialogs with descriptive text. Beginners use these to orient in new applications. */}
{/* - **Immediate:** Direct-manipulation controls with instant effect. No intermediary step required. */}
{/* - **Invisible:** Memorised commands with no visual locus — keyboard shortcuts, gestures. Expert users rely on these for speed, but they require prior knowledge. */}

{/* ## Working sets */}
{/* The subset of commands a user memorises through repeated use. Working sets are personal but overlap significantly among users with similar patterns. */}

{/* *Minimal working set*: The commands most users will need, identified through usage analytics or scenario-based design. Commands in the minimal working set should use immediate modality, not pedagogic. */}

{/* ## Trade-offs */}
{/* - **Layered interfaces** improve *initial* learnability and safety (preventing destructive errors) but may initially limit speed for tasks requiring advanced tools. */}
{/* - **Full functionality** offers maximum efficiency for experts but creates a "wall of complexity" for novices. */}
{/* - **Context matters:** Use progressive/layered approaches for infrequent users, novices, or high-risk tasks. Use full functionality for dedicated expert applications. */}

## Related patterns

### Mechanisms
- [Annotation](../?path=/docs/actions-sensemaking-annotation--docs) – marking content to add context and meaning
- [Explanation](../?path=/docs/actions-sensemaking-explanation--docs) – domain explanations support domain learnability
- [Transparent reasoning](../?path=/docs/activities-transparent-reasoning--docs) – showing *why* a system made a suggestion helps users learn the underlying logic
- [Progressive disclosure](../?path=/docs/actions-seeking-progressive-disclosure--docs) – structures the learning curve by hiding complexity
- [Bot](../?path=/docs/activities-bot--docs) – can serve as domain mentor, bridging system actions to domain understanding through contextual enrichment, question-driven exploration, and world knowledge integration

### Lifecycle
- [Onboarding](../?path=/docs/activities-onboarding--docs) – the initial spike in learning
- [Mastery](../?path=/docs/activities-mastery--docs) – the long tail of expert performance

### Foundation
- [Agency](../?path=/docs/qualities-agency--docs) – competence facilitates agency

### Tangentially related
- [Localization](../?path=/docs/activities-localization--docs) — learning curves might differ across localised interfaces
