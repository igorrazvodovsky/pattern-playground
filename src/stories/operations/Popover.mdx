# Popover

Present contextual content in a transient layer anchored to a trigger.

<Story of={ PopoverStories.Popover } />

## Variants

A popover's trigger model and dismiss behaviour follow from one variable: *is the content interactive?*
Interactive content forces click trigger, focus management, and explicit dismiss. Non-interactive content permits the lighter hover/focus model.

### Popover

- *Light-dismiss.* Clicking outside the popover closes it and returns focus to the trigger.
- *Focus management.* Opening the popover moves focus to the first interactive element inside it.
- *Keyboard.* Escape closes the popover and returns focus.
- *Anchor binding.* The popover is semantically connected to its trigger via `popoverTarget`.

### Tooltip

When the content is non-interactive, the popover operates as a tooltip — supplementary text revealed on hover or focus.

<Story of={ PopoverStories.Tooltip } />

- *Hover intent.* A short delay filters accidental hover — the layer appears only when the cursor lingers.
- *Focus trigger.* Keyboard actors reach the same information via focus, without requiring hover.
- *Non-interactive.* Tooltips contain only text. If the content needs to be acted on, use a click-triggered popover.

Avoid putting essential information only in a tooltip. Tooltips are hidden by default; actors who miss them must not lose access to critical functionality.

## Accessibility

Tooltip content must be reachable without a pointing device. Every tooltipped element needs a focus trigger so keyboard actors can access the same information.

Bind non-interactive popovers to their trigger using `aria-describedby` — this lets screen readers announce the content when the trigger is focused, without replacing the element's own label.

Interactive popovers use focus management instead: focus moves into the popover on open and returns to the trigger on close.

## Design considerations

Use popover when the contextual content is brief and self-contained. When the content requires a decision before the actor can continue, prefer [dialog](../?path=/docs/actions-application-dialog--docs). When the actor needs to work alongside the secondary content, prefer [drawer](../?path=/docs/actions-application-drawer--docs). See the [modality decision tree](../?path=/docs/foundations-modality--docs#overlay-vs-page) for guidance.
