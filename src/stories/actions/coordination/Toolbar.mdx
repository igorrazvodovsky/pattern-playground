# Toolbar

The persistent exposure of frequently used commands, balancing quick access with visual economy.

<Story of={ ToolbarStories.Toolbar } />

## Context

Options presented in the toolbar can depend on the context and state of the parent component. Things that are not applicable in the parent’s state can be hidden. For example, if the parent is empty the user will have different needs compared with a full state.

## Order

Things inside toolbars are consistently ordered so that users can easily find things across different features and applications.

Features should be sorted in order of importance. Secondarily, features should be organized in logical groupings for easier recall.

## Related patterns

- [Action bar](../?path=/docs/actions-coordination-action-bar--docs) — the transient counterpart; summoned by selection rather than persistent, and typically composes a toolbar internally for its action group
- [Priority+](../?path=/docs/actions-coordination-priority--docs) — handles overflow when toolbar items don't fit the available space
- [Command menu](../?path=/docs/actions-seeking-command-menu--docs) — a keyboard-driven complement; exposes the full command set on demand where the toolbar exposes a persistent subset
- [Unavailable actions](../?path=/docs/operations-unavailable-actions--docs) — the pattern behind hiding toolbar items that are inaccessible in the current context
