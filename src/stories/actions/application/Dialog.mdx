# Dialog

The interruption of a workflow to demand a focused decision, balancing urgency against disruption.

<Story of={ DialogStories.Default } />

## Scrolling
By design, a dialog’s height will never exceed that of the viewport. As such, dialogs will not scroll with the page ensuring the header and footer are always accessible to the user.

<Story of={ DialogStories.Scrolling } />


## Related patterns

Dialog is the canonical [transient mode](../?path=/docs/foundations-modality--docs) — it suspends the primary workflow to force a focused decision. See the [modality decision tree](../?path=/docs/foundations-modality--docs#overlay-vs-page) for when to use dialog vs. [drawer](../?path=/docs/actions-application-drawer--docs) vs. a separate page.

- [Drawer](../?path=/docs/actions-application-drawer--docs) — when the actor needs to maintain awareness of the primary context rather than being forced to resolve before returning
- [Popover](../?path=/docs/operations-popover--docs) — lighter, non-modal alternative for contextual content that doesn't require a decision
- [Notification](../?path=/docs/actions-coordination-notification--docs) — dialog is the most disruptive notification type, reserved for critical alerts that require action
