## ABSTRACT

The user interface is a central concept in human-computer interaction, but peculiarly fuzzy. This affects discussions of the fundamentals of our discipline and the positioning of our work. We propose a model of the user interface that consists of four elements: Devices, Interaction Techniques, Representations, and Assemblies (DIRA). We explain their roles in the user interface and discuss some associated concerns about evaluation and design. We then show how to use the model to describe the elements of user interfaces (with examples that include a menu, a fish-eye interface, and notifications) and to analyze the central characteristics of user interface paradigms (including tangible user interfaces and mixed reality). Finally, we discuss how describing user interfaces with the model can drive their design and evaluation.

## 1. Introduction

The concept of *user interface* (UI) has been central to HCI since the beginning of the field. It has been used since at least the 1960s (e.g., Bennett, 1972; Parnas, 1969). Many HCI textbooks contain entries related to the concept of UI, such as its levels (e.g., Moran, 1981), components (e.g., Foley et al., 1989), and styles (e.g., Shneiderman et al., 2016). However, they either address particular parts of UIs (e.g., input devices in Card et al., 1983) or associated processes (e.g., for design in Foley et al., 1996) instead of the user interface itself and as a whole.

The literature in general is burgeoning with research on different types of UIs; textbooks exemplify graphical, tangible, speech, and mobile UIs (e.g., Benyon, 2019; Preece et al., 2002; Shneiderman et al., 2016), and papers discuss UI paradigms from natural user interfaces (e.g., Macaranas et al., 2015; Norman, 2010; Wigdor and Wixon, 2011) to mixed reality (e.g., Milgram et al., 1995; Speicher et al., 2019). However, the fuzziness surrounding the concept hampers our ability to answer even the simplest questions about these user interfaces: How exactly do they differ? What characteristics make them distinct types, classes, or paradigms of UIs? And what are the essential components to address in research about them? To us, answering these questions seems unnecessarily difficult.

Our motivation for investigating the concept of UI is to help answer these questions in positioning and doing our work. For example, a researcher could position their work on ultrasound haptics or motion gains in virtual reality to particular parts of user interfaces while relating those to the UI as a whole and to other, existing UIs. A clearer concept of the user interface could also help generate ideas for research

on UIs of the future. For example, the design-space mappings of input devices in the 1990s by Card et al. (1991) helped to systematically go through possible designs, generate ideas for unexplored ones, and increase experimental control of the surrounding factors. We suspect that similar benefits would follow with clarity about the other constituents of UIs and the UI as a whole. The higher-level motivation is to help to discuss the fundamentals of our discipline. For example, it is hard to explain what exactly it is that we *study*, *evaluate*, and *design*. Having a clear concept for the UI would help describe HCI research in general.

We approach the investigation of the user interface by attempting to formulate a model of it. We use the term model as a deliberate simplification of actual user interfaces with the aim of making them more understandable. A model here serves as an idealization of the real world (Frigg and Hartmann, 2020) and help us to “gain understanding of a complex real-world system via an understanding of simpler, hypothetical system that resembles it in relevant respects” (Godfrey-Smith, 2006). We build on existing views, concepts, and definitions related to UIs to propose a new model that improves the conceptual clarity of the user interface as an entity and of its constituents.

In the following, we first discuss the extant attempts to capture what a UI is. We then propose a model of the user interface that consists of four elements: Devices, Interaction Techniques, Representations, and Assemblies (DIRA). We define these elements, relate them to concepts and models from the literature, argue for their roles in the UI, and exemplify design and evaluation concerns related to them. We show how to use the DIRA model in analyzing and describing user interfaces, and thereby illustrate how it can enable HCI researchers to:

## 2. Related work

The concept of user interface has been in widespread use since the late 1960s (e.g., Bennett, 1972; Parnas, 1969) and close synonyms (such as man-machine interface, user-machine interface) since the mid 1960s (e.g., Fenton, 1966; Nickerson, 1969; Shackel, 1969; Hornbuckle, 1967). Parnas (1969), for instance, noted that “by user interface we mean the aspects of the system behavior that a user sees” (p. 379). The emergence of this concept is intimately related to the history of ergonomics and HCI, which has been thoroughly reviewed in numerous places (Baacke et al., 1995; Grudin, 1993, 2017). Instead of tracing this history, we start by discussing common, higher-level views about user interfaces and next discuss work that has provided more in-depth definitions or models of the user interface.

### 2.1. Common definitions

With common definitions of the user interface, we mean those that appear in textbooks and reference works on HCI. All are places where a synthesized definition of the concept could be expected. We see a few varieties of explanations.

Several sources explain the user interface using close synonyms of interface. For instance, Wikipedia gives more than 4500 words of discussion of the concept (Wikipedia, 2019). In particular, it states that the user interface is “the space where interactions between humans and machines occur”. A textbook on 3D user interfaces by LaViola Jr. et al. (2017) suggested that the UI is “the medium through which the communication between users and computers takes place”. Card et al. (1983) noted that “the human-computer interface is easy to find in a gross way—just follow a path outward from the computer’s central processor until it stumbles across a human being” (p. 4). All of these definitions emphasize the *boundary* between human and computer. Although an intuitive definition, this implies an invisible, infinitely thin boundary and not an entity that one could analyze, design, or evaluate.

Other explanations of user interface focus on it as the totality of what the user experiences. Grudin (1993) argued that “the user’s interface to the computer may center on the software controlled dialogue, but it also includes any documentation and training that are part of using the computer. It includes colleagues, consultants, system administrators, customer support, and field service representatives, when they are available” (p. 114). Similarly, in a textbook on UI design, Lauesen (2005) wrote that “the user interface is the part of the system that you see, hear and feel” (p. 4). These definitions share a *holistic* view of UIs, including the user as a central element of the definition.

Others focus on the goal of the user interface. Raskin (2000), for instance, argued that “the way you accomplish tasks with a product – what you do and how it responds – that is the interface” (p. 2). This definition emphasizes the *actions* that are available to the user.

Finally, some textbooks contain *no* explicit explanation of the concept. For instance, the textbook by Hartson and Pyla (2012) contains three entries on user interface, but none of them defines or explains the term. Preece et al. (2002) list two entries under the index on user interface but do not define the term—in contrast, the book has a subsection called “what is a prototype”. Instead of defining the concept of UI, some textbooks give *examples* of styles of interaction (e.g., five styles of interaction such as direct manipulation exemplified with 3D interfaces or collaboration exemplified with face-to-face interfaces in Shneiderman et al., 2016) or interface types (e.g., the chronologically ordered 1980s command and graphical UIs, 1990s speech and touch UIs, and 2000s mobile, tangible, mixed reality, and many other UIs in Preece et al., 2002).

### 2.2. In-depth analyses of the concept of UI

The above sources address what a user interface is in general terms. Next, we survey some of the more specialized papers which discuss the UI in depth.

#### 2.2.1. Theoretical models in human-computer interaction

In 1981, Moran (1981) separated the user interface from other, hidden parts of interactive systems (which he calls implementations) in the Command Language Grammar (CLG). In his view, “the user interface of a system consists of those aspects of the system that the user comes in contact with—physically, perceptually, or conceptually”. The CLG consists of three components of a user interface, each divided into two levels. For example, in the conceptual component, the task level is about what tasks the user can accomplish with the system, and the semantic level describes the conceptual entities and operations in the system to accomplish the tasks (e.g., objects and their indication of functional capability). While the CLG is closely tied to a particular style of interaction – command-line interaction – it does move beyond the conventional definitions to provide an outline of the elements of user interfaces.

Foley has contributed a series of papers that discuss the user interface and its components (Foley, 1988; Foley et al., 1989, 1984, 1996); perhaps this is the most comprehensive work on the concept. The main idea is to separate four components of the user interface; two about meaning and two about form—this results in a “four-level user interface model, consisting of conceptual, functional, sequencing, and binding levels” (Foley, 1988). The conceptual component is about the principal application concepts that the user must learn. The functional component is about semantics, what may be done in the interface. Both of these are about meaning in the interface. The sequencing design (also called syntactic in some places) is about ordering inputs and outputs. The binding component (also called lexical design) is about the form of the UI, that is “how input and output units of meaning are actually formed from hardware primitives” (Foley et al., 1996, p. 395). Bleser and Foley (1982) distinguished only the syntax and lexical aspects of a user interface to create a specification language.

In 1990, Chignell (1990) presented an ambitious taxonomy of user interface terminology as “a possible basis for the eventual development of human-computer interaction as a science” (p. 27). The part of the taxonomy that interests us here is the “Basic Interface Model”. It suggests that the UI has seven fundamental components, including actions (of the computer application), behaviors (of the user), contexts, displays, effects, forms, and goals. Somewhat surprisingly, behaviors include both user navigation (i.e., interaction) as well as interaction styles (e.g., menus), whereas input devices exist in its own category. Chignell also distinguishes forms, which are “the models in which actions, effects, and displays are embedded” (p. 30). Compared to other work, this is a useful category that includes desktop metaphors and 3D interaction; Chignell does not, however, explain this category beyond a few lines.

Early versions of the book *Designing the User Interface* (until the fourth edition) described the object-action interface model (e.g., Shneiderman, 1997). This model separates objects and actions in users’ tasks from objects and actions in the interface. The former is about the universe of objects and intentions, whereas the latter is about metaphors that help users understand the UI and plan which actions to take onto the objects and in what order. The model separates these concerns and suggests that there are more or less suitable mappings between the task and the UI. However, it does not explicitly define the user interface.

#### 2.2.2. Models for building user interfaces

In 1978, Reenskaug created a model for designing software for user interfaces, which after some iterations with other scientists at Xerox PARC converged into three parts and gave the name it is the best known

as: Model-View-Controller, or MVC (Reenskaug, 2003). The first part, Model, remains outside the user interface, but it manages the UI's logic and the data used by the UI. The View presents information to the user, and the Controller can be used to interact with both the view and the model. Thereby, MVC directly touches at least two parts of the UI. Although Reenskaug explained that “The top level goal was to support the user's mental model of the relevant information space and to enable the user to inspect and edit this information”, MVC is a ultimately about software constructs, and leaves other components of UIs untouched, such as input devices.

PAC is a model related to MVC which attempts to bridge the gap between higher-level theoretical models and the models for implementing the UI (Coutaz, 1987). For example, Coutaz (1987) stated in their motivation for PAC that “High level theories such as GOMS and CLG offer a way to understand the nature of human-computer interaction and help organize the design process” and argued that from the implementer's point of view, such “theoretical models are too abstract”, while “engineering models come too late and implementation models are too functional. Theoretical models do not help much in the practical needs of the implementer who is not necessarily a knowledgeable person in the field of human-computer interaction. Engineering models are useful for the refinement of user interfaces not for their initial definition. Finally, implementation models organize functional pieces of software without integrating the 'substantial marrow' of theoretical models”. While PAC concerns the gap between UI implementation and HCI models, it says less about how the “substantial marrow” of HCI connects to the distinct elements of the UI.

Model-based user interface development approaches (e.g., the languages, tools, and environments, such as UsiXML, CAMELEON, and eSPACE) address the challenges emerging from developing a UI to a variety of end users, contexts, computing platforms, and programming languages (Meixner et al., 2011; Sanctorum and Signer, 2019). For example, the architectural model Seeheim includes a Presentation component that is responsible of managing input and output devices, while Arch is a model that includes the devices themselves in an Interaction Toolkit component (Beaudouin-Lafon, 1994). Therefore, they also include the parts that are in physical contact with the user.

Dix and Runciman (1985) motivated the role of such UI development models in connection with their PIE model by relating models to formal statements, which “can only be made in the context of some assumed structure of the relevant system” and do not translate from one system to another. As a solution, Dix and Runciman proposed abstract models and the principles that “can be used to derive, or at least to constrain, the formal specification”. They further add that “theorems about the abstract model should aid understanding of the interactions between various principles and hence aid design”.

In relation to user interface management systems (e.g., Myers, 1995; Beaudouin-Lafon, 1994), much work has also been published on formal specification and automatic generation of user interfaces (e.g., Paterno, 1999; Farooq and Dominick, 1988; Janssen et al., 1993). User Interface Description Languages aim to provide the formal specifications of UIs for developing them (see e.g., Appert and Beaudouin-Lafon, 2006; Jacob, 1986; Jacob et al., 1999). These languages use, for instance, syntax diagrams, state transition diagrams, flow diagrams, and also object-oriented approaches. For example, Jacob (1986) described how specification languages can be written for direct manipulation user interfaces and stated that “Such a specification should describe precisely the user-visible behavior of the interface, but should not constrain its implementation”. This essentially means that their focus is on the input/output parts of the software. Gajos and Weld (2004) automatically generated user interfaces based on the separation between the models of the interface, the user, and the device. They use these as input to an optimization procedure that adapts particular interfaces. Some formal methods, such as formal verification, also support analytical approaches in evaluating UIs (e.g., Bolton et al., 2013). For example, Campos et al. (2020) compared the effectiveness of tools

for analyzing UIs, and Harrison et al. (2015) compared interactive devices with the tools. Although these approaches and the motivation for models are relevant to the present discussion, they mainly focus on what Grudin (1993) called the “computer's interface to the user”.

### 2.3. Open issues

As briefly reviewed, common definitions offer relatively little in terms of a structured way of thinking about the UI. Earlier work from the 1980s and 1990s has presented more in-depth models. However, those have been tied to specific interaction styles (e.g., CLG). In 1995, Grudin (1993) noted that “the term 'user interface' came into use when system development was very different than it is now” (p. 112); that remains true today. Moreover, most of the earlier models are tied to dialogs and commands. It is unclear if they work for sensing systems (Bellotti et al., 2002), multiple users, or systems that place less emphasis on commands (Nielsen, 1993).

A few concepts are related to the user interface and will be discussed later. *Interaction* is an important concept that Beaudouin-Lafon (2004) has argued should be the focus of the field's attention, rather than user interface. Hornbæk and Oulasvirta (2017) discussed seven views of interaction. None of them includes the user interface, although several of the views (e.g., dialogue, transmission, tool use) implicitly involve a UI. For now, we assume that there is something to be said about the user interface which is not just about interaction—in the worst case, perhaps the UI may be defined fully in terms of interaction (preview: we think it cannot).

The various models also seem to clash in how much the UI encompasses (e.g., tasks, practice, manuals); they are also surprisingly unclear about what role computation plays in the interface (as pointed out for interaction, too; see Hornbæk and Oulasvirta, 2017). Finally, terms that are commonly used today (e.g., interaction style, interaction technique) are not easy to relate to the general concept of user interface.

Addressing these open issues, the purpose of our model is similar to that of the theoretical models: to serve as a thinking tool in human-computer interaction research. But instead of a high-level abstraction, it aims to recognize that human-computer interaction research often focuses on specific parts of the user interface and the related theoretical, empirical, and design challenges; similar to the implementation models that help implementers to focus on specific parts while recognizing their mutual relations. The elements of our model stem from those parts that we consider to relate to distinct concerns in terms of understanding, evaluating, and designing UIs, and therefore also separate focus areas in HCI research.

## 3. The elements of user interfaces

We propose a model of the user interface. Our objective was to model the UI as an entity that HCI research could analyze, design, and evaluate separately. The model consists of four elements of the user interface: Devices, Interaction Techniques, Representations, and Assemblies (DIRA). All four elements need to be present to make that entity: the user interface. Fig. 1 depicts the four elements of the model. The arrows represent the direct relationships between the elements that are crucial to consider in HCI research when addressing concerns about the evaluation and design of a particular element. Table 1 exemplifies the elements as well as such concerns.

As depicted, we do not include the user nor the back-end of the computer (e.g., what earlier models referred to as an “application” or a “functional core”) within the UI, but similarly to implementation models, they are connected to the elements of the UI (e.g., user's input on devices and the computer's on assemblies). We refer to these entities simply as the “user” and “computer” when explaining their connections to the separate entity of the UI in the rest of the section.

### 3.1. Devices

Devices *sense* the user and *display* representations. Sensing here means that physical information from the user is converted to digital form so that it may be used to operate the assemblies (through interaction techniques). Such information can be the user's button presses, gaze, speech, or hand gestures. Displaying here refers to converting digital information from the computer (through representa-

tions) into physical forms, such as pixels, that the user may perceive. In this way, devices define the physical points of contact between the user and the UI. A UI is often made up of many devices. For example, motion sensors, buttons, touchpads, displays, vibration motors, and speakers may appear in different compositions in mice, keyboards, touchscreens, or mobile phones. With the physical contact points, devices also define the modalities that the user can use to interact with the UI. For instance, some devices enable sensing gestures, others touch; some devices allow displaying visual information, others haptic information.

#### 3.1.1. Concepts related to devices

The earlier architectural models include components related to devices. For example, PAC (Coutaz, 1987) and Seeheim included a Presentation component that is responsible for managing input and output devices, while Arch is a model that included the devices themselves in an Interaction Toolkit component (Beaudouin-Lafon, 1994).

Foley et al. (1996) also considered input devices as one of the essential elements of UIs and Card et al. (1983) argued that physical devices are one of the mechanisms constituting the interface. Moran (1981) further separated three components of the user interface, one being the physical component consisting of the device level and the spatial layouts of the devices and the graphics. For Foley et al. (1996), the devices include locator, keyboard, valuator, and choice devices.

Card et al. (1983) mentioned keyboards and displays as devices, and also detail “the basic components” of devices, such as buttons and video displays. In the Morphological Analysis of Input Devices, Card et al. (1991) illustrated how the components of the input devices connect. For example, a merge composition means that two devices are connected in a way that the resulting input space is a merge of those, such as when two one-dimensional slide sensors of a mouse allow input on a two-dimensional space. Layout composition means that two types of device are physically integrated in a device, such as buttons and a location sensor on a mouse.

We use the terms *sensing* and *displaying* for particular reasons. Earlier work discussed devices but did not separate sensing and input. For instance, Mackinlay et al. (1990) wrote that “[s]imple input devices are described in terms of semantic mappings from the transducers of physical properties into the parameters of the application”. Since then, context-aware computing and sensors in many forms have become part of UIs. Thus, we separate the issues involved in sensing from the issues in mapping what is sensed to operate the computer (the domain of interaction techniques). Instead of output, we use displaying, because the devices give the user, through representations, many kinds of information. This includes what could be termed output (e.g., displaying a desktop with its icons) but also notifications (e.g., stemming from the computer) and feedback (e.g., a moving cursor). With these two terms, sensing and displaying, we recognize that there are different kinds of devices. However, they are treated as the same element, because the concerns for their evaluation and design are similar.

#### 3.1.2. The role of devices

In our view, a UI must be able to sense the user; otherwise it is not a UI because the user cannot interact with it. In this view, a passive display showing the current weather does not alone make a UI because the sensed information, the weather, is not about the user. Moreover, a UI needs a device to display to the user what we call representations. An implantable device that tracks only features of the user’s body is not a UI if those features are not displayed to the user. And even if, for instance, a heart rate is sensed and displayed, these devices do not alone make a UI if the heart rate is not sensed for interacting with the UI and through that to use a computer (as per the requirement of all elements being present to make a UI).

However, a UI does not need to sense a user’s input at all times, nor does it need to be limited to displaying things only for the purpose of interaction. For example, a camera may both be part of a UI and sense information not used in the UI. And a display device can show a background image which is not a representation used in interaction, but which can still influence the user experience of a computer as a whole.

#### 3.1.3. Evaluation and design of devices

From the essential roles of sensing and displaying follow two concerns for developing devices for UIs. The first one is about whether a device can sense and display suitable information for the user to interact with the UI. Similarly to Card et al. (1990), we call this the *expressiveness* of the device. The second one is about how well a device can sense the user and display information back to them. This is a matter of degree, rather than whether information can be expressed at all. Similarly to Card et al. (1990), we call this the *effectiveness* of the device.

In the context of visualizations, Mackinlay et al. (1990) described expressiveness as follows: “A human-machine artificial language can be used to express some information when it can encode exactly the input information, that is, all the information and only the information”. Card et al. (1991) and Mackinlay (1986) further argued that a problem in expressiveness of an input device arises when the input set does not match what can be expressed to interact with UI. For example, if the UI contains values that the input device does not sense, the user cannot express all the legal values needed to interact with all its functionalities.

And if the input device contains values that the UI does not display, the user can express illegal values. Expressiveness in our usage concerns sensing range, such as whether information from the user (say, pose, heart rate, eye gaze) may be sensed; and whether a specific sensory modality (say, olfaction, shape-change) is available to be displayed. For effectiveness, Mackinlay et al. (1990) suggested that the key is how good a device is at “exploiting the capabilities of the output medium and the human visual system” (p. 111). For example, a very high resolution or frame rate of a display does not necessarily increase how effectively the user can interpret a representation, and thus wastes computing power, while too low rate may hamper interpretation.

The key measures for expressiveness and effectiveness are spatial (e.g., resolution) and temporal accuracy (e.g., latency). The concerns in the spatial domain are about the modalities and ranges for sensing and displaying. For example, a concern can be whether a motion sensor’s range and resolution can detect the gestures the user wants to express. The concerns in the temporal domain are about latency, such as sampling and frame rate. For example, the user can experience latency in receiving feedback from their movements in virtual reality due to delays in either motion sensing or in rendering their avatar on the head-mounted display.

### 3.2. Interaction techniques

Interaction techniques map what is sensed by the devices to *operations* on the assemblies. Such operations include, for instance, pointing, selecting and manipulating objects, and navigating. For example, the operations of turning a lamp on with the switch and seeing the feedback that it does, or grasping a virtual object and receiving haptic feedback from the vibration motors of virtual reality controllers involve an interaction technique.

Thereby, interaction techniques map what is sensed by the devices to what happens in the UI. What happens in the UI is, in turn, determined by the assemblies and perceived as the feedback from the representations displayed on the devices. For example, an interaction technique called the control-display ratio is what maps the physical movement of the mouse (a device) to the position (an assembly) of a cursor (a representation) on the display (a device).

An interaction often consists of many operations in a sequence. For example, point-and-select, or a double click on an icon, are common sequences with graphical user interfaces. Also keys are often used in a sequence. For example, pressing a key with character “k” to type that character is an operation, but an interaction technique can also involve a combination or a sequence of command keys together with letter keys. These can help the user to efficiently type capital letters, move in a document, and give more complex commands or use shortcuts.

#### 3.2.1. Concepts related to interaction techniques

The notion of control relates to interaction techniques. In the PAC model, Control is a link between Abstraction and Presentation (Coutaz, 1987). In Seeheim the Dialogue control is similarly about managing “user input and application commands” and “application data and graphical objects” (Beaudouin-Lafon, 1994). Interestingly, Seeheim (1) unifies user input and graphical objects under the same notion of Dialogue control, but yet discusses them separately, and (2) mentions Application (commands and data) in connection with these both. In our view, this role of managing, controlling, or mapping of “User input and application commands” relates primarily to Interaction techniques (with connection to Devices and Assemblies) and the role of “Application data and graphical objects” to Representations. Their Application itself, in our view, resembles Assemblies.

For Foley et al. (1996), interaction techniques are “a way of using a physical input/output device to perform a generic task in a human-computer dialogue”. Foley thus connected devices to interaction techniques but did not exemplify how devices would work as part of interaction techniques. The Command Language Grammar

by Moran (1981) contains a Communication component with an Interaction Level. Moran explained that “the Interaction Level describes the dialog structure”, and that the dialog is “resolved as a sequence of physical actions—key presses and other primitive device manipulations by the user and display actions by the system”. Thus, Moran also connected devices (i.e., physical actions like key presses) to interaction techniques (e.g., sequence of those presses), but in our model the “display actions” are determined by assemblies.

LaViola Jr. et al. (2017) (pp. 79) suggested that

[i]nteraction techniques are particular elements of a UI. They are the methods by which a user accomplishes a task or part of a task, and they are primarily concerned with the form of the input given by the user (although they also involve display elements that give feedback to the user), and the mapping between that input and its result in the system.

This description also assigns mapping as the role of interaction techniques, but in the DIRA model the result in the computer and the display of feedback from those results are allocated to assemblies, representations, and devices.

We also use the term *operation* instead of the term task. Bødker and Klokose (2011) separated operational aspects in their human–artifact model. According to them, the operational aspects are about *how* the user’s actions are realized (with an interactive artifact). Similarly, we distinguish operations from the tasks users want to do and the activities they are engaged in; those are not part of the user interface but of the world surrounding the UI. Thus, the term operation also means that operations are small steps (working on assemblies) of using the computer to accomplish an entire task.

#### 3.2.2. The role of interaction techniques

In our view, a UI needs to have the ability to use sensed information from the user to operate the assemblies and through that, to use the computer. For example, seeing oneself moving on the map of a mobile phone involves an interaction technique that converts the user’s physical position in the world to the position of the blue dot on the display as governed by the assemblies. However, this position information is not necessarily passed to the computer by the assemblies (unless the position is used, for instance, to notify about a route for navigation).

In addition to enabling operations, interaction techniques aim to *help* perform those. For example, predictive text entry that autocorrects and completes words typed on an SMS, or a snapping technique that snaps a cursor to an estimated target icon, are interaction techniques using some kind of intention prediction. Intention prediction or classification (whether it is words or pointing targets) are in turn algorithms that use models about the user (e.g., learned models of user behavior, such as frequently used words or applications) or models about the task context (e.g., frequently used words in the entire language in general, or in this context, such as with a specific messaging application). The interaction techniques are designed to use such models to help the user perform the operations effectively (e.g., input the intended letters or hit the intended target).

#### 3.2.3. Evaluation and design of interaction techniques

The central concern for interaction techniques is user performance, that is, how well interaction techniques support operations. For example, control-display ratios of a mouse can be optimized to improve the user’s performance (e.g., speed and accuracy) in a point-and-select operation, and transfer functions on the user’s motions in virtual reality can be tuned to improve usability and user experience.

Another central concern is feedback. Because interaction techniques determine how the user’s actions are mapped to the operations, they also directly influence how users, as a result of their actions, receive feedback. Feedback in turn influences learning. For example, adjustments in C-D ratio influence how the user receives feedback of their

mouse movements and thus how fast they learn to point with a new C-D ratio. It also influences the sense of control. For example, dwell time in gaze selection determines when the confirmation is given, and hence the feeling of being in control of the computer. Finally, feedback influences physical responses. For example, the mismatch a joystick walking technique in virtual reality creates between the actual and perceived motion induces simulator sickness.

### 3.3. Representations

Representations *embody* the user and the computer. Representations of the user include, for instance, a cursor on a desktop and an avatar in virtual reality. Representations of the computer show, for instance, icons of apps on a mobile phone or text on a document file. Representations can embody the user or the computer in any modality. For example, the user may hear which objects they can interact with and be told which oral commands they can use on a speech interface.

The user representations may directly show the user (e.g., as a cursor), and hence explicitly indicate their location in the UI (which is given by the assemblies, changed by interaction techniques, and displayed by devices). The location can also be implicit on the UI, such as when the path is shown on a command line as text representations or when the current view on a mobile phone’s touchscreen shows the user can tap any of the visible icon representations. Finally, multiple user representations can be shown at once. For example, a desktop user can see a blinking cursor on a text document wherein they can type, and a separate mouse cursor with which they can point.

#### 3.3.1. Concepts related to representations

Representations are related to Instruments by Beaudouin-Lafon (2000). Instruments in Beaudouin-Lafon’s view include, for instance, scrollbars, cursors, and text. Beaudouin-Lafon argued that the instrument contains a logical part which defines “the representation of the instrument in software and on the screen” (p. 448). In our model, the representations are about embodying the user and the computer on the devices, whether visually on a screen or via other modalities. However, their counterpart in software, such as where they appear and how they move, are defined in assemblies and interaction techniques, respectively.

Seinfeld et al. (2020) defined user representations as

artificial extensions of users’ physical bodies, enabling them to execute actions in virtual environments, which otherwise would be unreachable. For example, in a traditional desktop environment, users select and deselect different virtual objects viewed on a screen, through the use of a cursor, but not using their hands directly to touch and grab these virtual objects. When playing a desktop video game, users are frequently represented by an avatar that performs different actions in the virtual environment to play the game.

Seinfeld et al. (2020), however, did not explain the role of user representations in UIs where the user is not directly visible, such as speech, touchscreen, or tangible interfaces. And while they stated that the continuous control result “in a perceived sense of agency over the representations” (p. 403), the sense of agency in our view is facilitated by interaction techniques, because they are responsible for the operations (e.g., a change in the position of the representation) the user may get feedback from.

Text is also a representation, as Chapans (1965) discussed already in 1965, stating that “... changes in the words used in man-machine systems may produce greater improvements in performance than human engineering changes in the machine itself”. In our model, text may represent both users (e.g., the text describing the user’s location in a directory) and the computer (e.g., the text describing menu item names).

#### 3.3.2. The role of representations

A UI needs to have the ability to convey the states, and changes in the states, of the user and the computer. The UI conveys this through representations. Therefore, representations also reflect how the computer can be used (similar to Beaudouin-Lafon's instruments). For example, a cursor is a user representation – where am I in space – which often also represents the state of the computer and thereby affords certain types of operations. When the computer representation is text, the cursor may take an I-form for selection or positioning; when it is a map view, the cursor may take an arrow form for panning; and when it is a virtual button, the cursor may take a hand form for pressing. Representations also change to provide feedback to the user. For example, an icon might be highlighted when it is clicked, or the user may receive vibration feedback to confirm a tap on a touchscreen. In summary, the representations are responsible for illustrating the state and changes in the state of the user and the computer on the devices, while the state is given by the assemblies and influenced by interaction techniques.

#### 3.3.3. Evaluation and design of representations

Representation – whose function is to embody the user and the computer – gives raise to a few central concerns. The relation, or match, between representations and the users' world is crucial. It has given rise to the notion of semantic distance (Hutchins et al., 1985) between how the user construes the world and how the world appears in the UI. The use of metaphors in the UI (Barr et al., 2002) also relates to how representations are designed. Both the semantic distance and the metaphors aim to comply with the requirement of the user being able to recognize representations (e.g., icons, command names). For example, an icon's reflection of the file's or an app's type is one of such concerns. Another is the state of the computer which the representation reflects. A UI can have representations that are inactive (i.e., that cannot currently be interacted with). For example, a menu can have grayed-out items, and a view the user sees through augmented reality glasses can include real objects that cannot be interacted with. The concern here is how to show whether a visible (or otherwise perceivable) item *affords* or does not afford interaction.

### 3.4. Assemblies

Assemblies *organize* the representations and connect the operations from the interaction techniques to the computer. Examples of assemblies include the arrangement of files in a directory, the layout of icons on a desktop, and the placement of content in a browser window. Although the assemblies connect to the computer, they do not necessarily follow the organization of the computer. For example, a directory links the folder and file icons (representations) to the computer (e.g., content, data structure). Yet, what is seen as the organization of representations is not necessarily similar to the computer's organization. For example, some files may be omitted from a directory view (e.g., to safeguard a user from editing files related to the computer's functionality).

Assemblies define how representations are organized on the UI both spatially and temporally. In the spatial domain, an assembly specifies, for instance, the organization of menu items (e.g., the order of the items within each menu level), or of a file directory (e.g., an alphabetical order based on the file name or a chronological order based on the date when the user last modified the files). Assemblies are often spatially nested. For example, the organization of a menu is determined by assemblies for both the levels and the organization of items within the levels. Similarly, a browser assembly can organize windows in tabs, and another can organize each window; a desktop assembly can organize desktops, and another the layout of icons therein.

Often, organizations involve a temporal dimension in addition to a spatial one. For example, in non-visual modalities, such as in a speech user interface, menus and directories can be conveyed with audio in a temporal order as specified by an assembly. The assemblies also specify how representations will react to changes as a result of interaction techniques. For example, when a file icon is selected by an interaction

technique, the assemblies connect this to a temporal feedback through the representation (e.g., flashing the icon as immediately as possible), and connecting the file in the computer to showing its representation (e.g., opening a Word-document, say, a second after). Finally, assemblies specify when to show the user when an event occurs in the computer. Notifications, for instance, appear as the results of principles about which changes in the computer should be shown to the user, such as when a notification badge appears on an email app icon.

#### 3.4.1. Concepts related to assemblies

Concepts similar to those of assemblies already exist in the HCI literature. For example, assemblies relate to the Model of MVC (Reenskaug, 2003), to the Abstraction of PAC (Coutaz, 1987), and to the Application of Seeheim (Beaudouin-Lafon, 1994).

In several publications, Beaudouin-Lafon and colleagues have used a related notion of substrates (e.g., Beaudouin-Lafon, 2017; Maudet et al., 2017). For instance, the description of what Maudet et al. (2017) call “graphical substrates” links the substrates directly to representations. They explain that substrates “guide the layout, but rarely appear in the final result” and that “changing the substrate usually affects the layout as well”. In the latter quote, the layout can be thought of as a layout of representations. The graphical designers who participated in the study of Maudet et al. (2017) called graphical substrates “systems, principles, architectures, structures, rules, and constraints”. Based on the input of designers, Maudet et al. (2017) presented a descriptive framework for graphical substrates. The substrates were created based on concepts (such as a landscape of letters or an “ambience”), content properties (e.g., content of the same nature has the same style, such as titles and subtitles), and context constraints (e.g., page dimensions). These are mapped by the substrates to “output properties”, which include spatial and temporal properties. This is consistent with our view of the assemblies, but does not elaborate on the role of representations. In other work, Beaudouin-Lafon (e.g., 2017) has defined substrate as “a digital computational medium that holds digital information, possibly created by another substrate, applies constraints and transformations to it, reacts to changes in both the information and the substrate, and generates information consumable by other substrates”. In our usage, the term assembly is mainly about organization of representations in space and time, rather than a unifying principle of interaction.

Assemblies also relate to the object-action interface (OAI) model by Shneiderman et al. (2016). According to the OAI model, objects and actions can be at different levels of abstraction or complexity. At the interface side, this might concern metaphors and plans at the highest level of abstraction. This is similar to assemblies because it concerns the fundamental organization of the UI. However, the OAI model also groups concerns related to pixels and clicks to a lower abstraction; for us, they concern representations and interaction techniques. In contrast to the task side of the OAI model, our model does not include the user's higher-level action plans and the user's learning of object properties within the UI, but rather consider them as design concerns related to the UI.

#### 3.4.2. The role of assemblies

In our view, assemblies organize where and when to show which representations. Without principles or rules for this, the representations might not be shown at all. Similarly, assemblies also require representations, since the assembly itself is a set of principles and rules that are not perceivable to the user without representations. For example, an alphabetical order of items in a drop-down menu would not exist for the user if the items were not perceivable.

#### 3.4.3. Evaluation and design of assemblies

From the essential role of assemblies – to organize – several high-level concerns for their evaluation and design follow. One is that

the organization of the UI (or more precisely, of representations) should *match* the user's tasks. According to work on information architectures (e.g., Morville and Rosenfeld, 2006), matching the user's tasks could happen, for instance, through the structure of information (e.g., hierarchical, linear, topical) or the sequencing within that structure (e.g., alphabetical, chronological). The correspondence in organization of the UI and users' tasks can be investigated with methods such as card sort, which derives organizations from users, or information scent, which assesses the probability that users will navigate a hierarchy correctly, given a task (Pirolli, 1997). A similar idea is captured in aligning the mental models of designers and users of a system (Norman, 2013). Expressed in the vocabulary of the present paper, such alignment means that the spatial layout as well as its structure in steps and time should be as similar as possible to the user's tasks in the real world.

Another set of concerns for assemblies is to relate to the physical characteristics of devices and representations. For example, a web page's fluid design is an assembly that organizes its content (representations) differently based on the screen, and thus the browser window shape and size, such as a larger and horizontally wider desktop display versus a smaller and vertically longer mobile phone screen. Similarly, mixed reality uses assemblies to lay out virtual content (e.g., objects, texts) over the view of the physical world, and this organization needs to adapt based on the physical world. For example, text annotations of sights need to align with those sights independent of the user's head orientation around them. In our terminology, the idea of responsive design is to create an assembly, in particular rules, that capture which representations to use and how they should be organized, given a particular device.

Yet another concern for assemblies is to improve the discoverability of representations. For example, with menu and directory organizations this can be about depth versus breadth, and organization within levels. Some interaction styles, like direct manipulation, allow the user to directly act on the objects, which requires objects to be visible. Visual attention can also vary across different parts of a display device or in relation to other representations, and thereby considerations include whether to place important items on the left of web pages.

## 4. Using the model

The aim of the DIRA model is to improve the conceptual clarity about user interfaces and help describe, analyze, and evaluate them. In this section, we first show how to use the model to describe simple UIs (Section 4.1). We then show how to use it for analyzing the central characteristics of UIs (Section 4.2). Finally, we discuss how to use the model in the design and evaluation of UIs (Section 4.3).

### 4.1. Describing user interfaces with the model

We start by showing how to use the model for describing the elements of a menu, a fishye interface, and a notification. These simple UIs were mentioned in connection with the individual elements above, but here we focus on them as a whole and disentangle the kinds of devices, interaction techniques, representations, and assemblies they contain.

#### 4.1.1. Elements of a menu

Drop-down menus are perhaps the most commonly used menus on computers (see Fig. 2). The devices involved in using a drop-down menu can consist of a display, a keyboard, and a mouse.

The main operations with menus are to move (as part of navigation) and to select. With a keyboard, the interaction technique to move in a menu can involve, for instance, pressing the shift, arrow, and return keys. The shift (or arrow) key press can move the user horizontally to the zeroth level of the menu, opening different dropdowns. The arrow keys can then be pressed for moving up and down among the items of a dropdown, and the return (or right arrow) key to either open another

level of the menu (a new dropdown next to the first level item) or to select an item if there are no more levels underneath the item.

The menu items are representations, as is the box containing the items. The representations of menu items are usually text labels. When moving in the menu with the mouse or key presses, the representations change to give the user feedback by highlighting the item representation (e.g., by blue) at which the user is currently located.

The assembly defines the organization of the items in the menu, its levels, which items are shown, and how they are shown (e.g., current, available, or grayed-out). Thereby, the assemblies also determine when a representation shows that a new level is available on an item and when the new level is shown upon selecting such an item.

#### 4.1.2. Elements of a fishye interface

Fishye interfaces manipulate the view based on the user's location on the user interface (Fig. 3). It shows parts of the view in high magnification (i.e., focus) and other parts in low magnification (i.e., smaller). The devices of a fishye interface can be a display, and a touchpad, a mouse, or other positioning device.

With fishye interfaces, the part of the view that is in focus is determined by interaction techniques. Although the movement of the focus is a crucial characteristic of the fishye UI, the interaction technique is similar to conventional control-display functions.

The representations in fishye UIs are a cursor position or a position on a scrollbar (i.e., user representation). The representations that the fishye manipulates are, for instance, icons or text labels on a list or a menu. The fishye menu generally keeps these representations in their original appearance, only changing their size or distorting them.

The assembly determines how this distortion is applied; the amount of magnification, a possible decrement in size on the edges of the view, and how gradually this change is applied. Assemblies also determine the layout of the items in a set (e.g., icons on a list) and which part of the list is shown. In principle, the entire list can be in view at once because the fishye can make the items on the edges infinitely small while allowing the user to still see the items in focus.

#### 4.1.3. Elements in notifications

Notifications include badges, banners, sounds, and pop-ups. Notifications are delivered to the user by devices, such as visual displays or speakers, and can involve devices for reacting to them. Let us use a mobile phone with a touchscreen display as an example device.

The interaction techniques that help the user to react to a notification may be, for instance, taps and swipes to open the phone and select the notification (e.g., a badge on an app icon, a banner on the screen), and selecting the reaction (e.g., opening an SMS with a tap, or dismissing the notification by swiping it away).

The representations can be banners on a locked screen, an icon for the app that the user is notified about, a badge presenting a number of new messages on a messaging app, and a sound played for each new message.

The assembly determines whether and when the notification is played (as a sound, or displayed as a banner or a badge). It also determines the layouts of the banners on a notification center, and the layouts of the icons and the views.

### 4.2. Analyzing user interfaces with the model

One aim of the model is to analyze existing and new UI paradigms in terms that capture their central characteristics. Preece et al. (2002) stated that a UI paradigm refers to the “shared assumptions, concepts, values, and practices” with which the community of researchers and designers approach the paradigm. Therefore a paradigm should reflect some particular characteristics of the UI.

In this section, we show how the DIRA model can be used to re-interpret four central and debated UI paradigms and to highlight their key characteristics.

**Tangible user interfaces** (TUIs) were first envisioned by Ishii and Ullmer (1997). Subsequently, the paradigm has been discussed in many works (e.g., Fishkin, 2004; Ishii and Ullmer, 1997; Ullmer and Ishii, 2000). The core idea is often presented by contrasting it with graphical user interfaces (GUIs). For example, Ishii and Ullmer (1997) explained that “GUIs represent information (bits) in the form of pixels on bit-mapped displays. These graphical representations are manipulated with generic remote controllers (such as mice and keyboards)”. However, “[r]ather than make pixels melt into an interface, TUIs use physical forms that fit seamlessly into a user’s physical environment”. Thereby, “[t]he key TUI idea remains: give physical form to digital information, letting serve as the representation and controls for its digital counterparts”. To support this idea, Fishkin (2004) presented a taxonomy and analysis of TUIs through embodiment and metaphor axes. This view helps to analyze the degree of tangibility of UIs.

Instead of representations, however, our model highlights devices and interaction techniques as key elements of TUIs. In our model’s view, TUIs stand out in terms of *devices* from other UIs, which “bias towards graphical output at the expense of input from the real world” according to Ishii and Ullmer (1997). In Ishii and Ullmer’s metaDesk TUI, the user controls a scene on a table-top display with physical objects. These physical objects are called Phicons and can be, for instance, miniature buildings. The Phicons are in our view devices (which are used for sensing similar information than buttons, knobs, or mouse), when combined with a graphical display. The Phicon buildings can be placed on the display surface to show and manipulate a map scene, such as by panning, scaling, or rotating. These are the *interaction techniques* characteristic for a TUI. In contrast, the representations and assemblies used in TUIs demonstrate no unique characteristic compared

to other UIs with displays (e.g., GUIs) and physical objects (e.g., mixed reality).

**Natural user interfaces** (NUIs) (Wigdor and Wixon, 2011) are operated through intuitive actions based on natural, everyday human behavior. However, the term is unclear, as discussed by many (e.g., Norman, 2010; Macaranas et al., 2015). The way we see natural UIs is shaped by the four elements. One aspect of natural comes from what is sensed (by devices), such as multi-touch interaction or gestures. Another aspect of natural is related to interaction techniques, which should be based on the user’s familiar, everyday bodily movements and thus require little learning. In this context, the issue with gestures is that developing device-technique pairs that are both expressive (e.g., the resolution of sensing multiple gestures) and learnable (e.g., performing all the gestures accurately) is difficult. This is similar to the critique by Norman, who argued that such interfaces are not natural because their interactions, say gestures, are not easy to learn or remember (Norman, 2010). Therefore, the key elements characterizing TUIs and NUIs in contrast to other paradigms appear to be the devices and interaction techniques.

**Mixed reality** (MR) and its relation to **virtual reality** (VR) is discussed in many papers from Milgram’s Reality-Virtuality continuum (Milgram et al., 1995) to a paper at CHI 2019 on MR by Speicher et al. (2019). Speicher et al. (2019) build a conceptual framework of MR with seven dimensions: the number of environments, the number of users, the level of immersion, the level of virtuality, the degree of interaction, input, and output. In Speicher’s dimensions, the input includes “in a more general sense anything that sensors can track” (Speicher et al., 2019). Output is given “to one or more of the user’s senses in order to change their perception”, that is, a stimulus to sensory modalities. Output is often visual, but Speicher et al. argued that it can

Speicher also lists spatial registration as a special feature of MR. It is about sensing the physical environment with devices to map content there. While this is an interesting observation of features of MR, it is not exclusive to it. Sensing UIs from TUIs to Ubiquitous Computing to VR needs it, too. VR, for example, needs spatial registration for the tracking space. Unmapped physical objects would likely be dangerous for the user who could collide or stumble on them. Mapped physical objects in VR are also increasingly used for providing haptic feedback as such, or with redirections of the user's motions. Thus, in our view, the devices and interaction techniques of MR do not make a clear distinction from other UI paradigms.

Instead, our model highlights the representations and assemblies as key elements of MR. MR has two prominent characteristics with respect to *representations*. First, computer representations can take two forms: they can be of fully virtual content, such as objects (similar to VR or GUIs), or augmentations of reality, such as physical objects (similar to TUIs). Second, the user representations in MR are implicit (the real body), whereas in VR those are usually explicit avatars and with other UIs less body-oriented and more device-oriented, such as cursors.

MR can also be characterized by three key features of its *assemblies*. First, an MR scene always contains real objects that cannot be interacted with. In contrast, the entire visible scene of VR or GUIs is a more controlled and constructed assembly, and with TUIs the entire UI is centered around actionable objects. Second, as the physical surroundings influence the assemblies, other people present may change them by manipulating the surroundings and objects therein. Moreover, those other people might not be users of the same UI for MR but only bypassers. Such concern of bypassers changing an assembly is small with other UIs, as even with VR and TUIs the objects are attended mostly by users only. Third, co-location of users combined with the number of users in the same environment (also one of Speicher's dimensions) pose distinct problems and opportunities for MR and VR. In MR, the co-located users inevitably share the same physical part of the surroundings (number of environments) as well as awareness of each other's presence (number of users). In VR, physically co-located users can act in completely distinct virtual surroundings. Moreover, the nature of co-location of other users may take three forms that are not possible for MR: (a) VR users can be unaware of other's being in the same physical environment and even influencing their virtual world, (b) VR users can be aware of others, but not know the nature of those others; the avatars could be characters in a game, other users, or artificial agents (e.g., AIs or conversational bots), and (c) VR users can be physically in different locations but virtually in the same environment, sharing the same view of the virtual surroundings, and acting in it through avatars. Therefore, the key elements characterizing MR and VR in contrast to other paradigms appear to be the representations and assemblies.

### 4.3. Use of the model in evaluation and design

The DIRA model can help to form structured concerns and questions about UIs. Some concerns were discussed in connection with each element. Next, we show how the elements can help in identifying concerns and questions for designing and evaluating UIs.

For the purpose of explaining how the elements of the DIRA model can be used in evaluation and design, we have summarized three types of UIs in [Table 2](#): Command Language Interfaces (CL), Graphical User Interfaces (GUIs), and Speech UIs. The idea is that summarizing UIs this way allows using the rows of the table, that is, the same elements of each UI, for comparison.

#### 4.3.1. Using the elements in design

The elements can be used to focus UI design on particular qualities of interaction. This can be done by attending to how the element is

realized in a UI and to which associated concerns of that element are supported by the design of that UI.

For example, *Command Language (CL)* on [Table 2](#) is a classic type of UI. In CL, representations might not be visible by default; the current folder name can be represented in front of the cursor for text input, but not the entire path from the drive to the current location. Consequently, assemblies are also not perceivable. A prevalent design concern compared to, for instance, GUIs is therefore discoverability (e.g., which files exist and in which folders).

Graphical User Interfaces (GUI) are a mature and widely used UI paradigm. GUIs are often defined through visibility, such as a system that “makes everything relevant to the task visible on the screen” ([Kimball and Harslem, 1982](#)). The Xerox Star UI was one of the first GUIs and considered to have the desktop as “the principal Star technique for realizing the physical-office metaphor” ([Kimball and Harslem, 1982](#)). Consequently, central design concerns include metaphors and recognition of graphical representations. In contrast, the text representations in CL can be more explicit and thus easier to match to the user's world.

The third paradigm in [Table 2](#) are speech UIs. Speech UIs are similar to CL in that assemblies are hard to perceive due to the lack of visual representations. Therefore, discoverability is a concern in speech UIs too. However, the interaction techniques of a speech UI (e.g., using natural language processing), may help the user overcome the invisibility and the resulting need to learn commands, perhaps easier than with CL (e.g., memorizing shortcuts).

#### 4.3.2. Using the elements in evaluation

The elements of UIs can be used as a thinking tool in empirical and analytical evaluations. In particular, the elements can help ensure and justify control of variables within UIs. For example, when comparing two UIs, they can first be aligned by describing them through elements similar to [Table 2](#). If the purpose of an experiment is to test, for instance, whether the evaluated UI improves selection speed with a new cursor gain (an interaction technique), then the other elements (i.e., rows on the table) of the UI except for the interaction technique should be kept similar. Thus, the evaluation should aim to use the same devices (e.g., a mouse), same representations (e.g., the cursor and the target object for selection), and same assemblies (e.g., the layout of selection targets).

The elements that require the most attention in evaluations are those immediately related to the measure. Sometimes, as with a cursor gain technique, this alignment is trivial. Occasionally, however, it can be tricky, such as when the tested element (e.g., technique) needs a device that differs from an available baseline (e.g., a different tracking device). In these cases, the model can help find suitable UIs for comparison by relating the elements in question to each other. For example, on-skin UIs can be used for tapping and pressing input. However, comparing interaction techniques for these is tricky, as the available baselines are touchpad or button-based input. The touchpad has a rigid surface for tapping and the button is pressed, but has a mechanical limit for actuation point; the skin has neither. Now, the on-skin techniques and the corresponding devices can be found more easily with the model. In this case, for instance, the technique could be tapping, and the related device to would be a touchpad (whereas for pressing it would be a button). The other elements (representations and assemblies) should be kept the same.

In evaluation, it makes sense to separate the success levels of the elements from each other. For example, at the device level, success might be about sensing accuracy. For interaction techniques, success is about user performance. For representations, it may be about users' ability to recognize an icon or a label. For assemblies, success might be about a match to the user's tasks. When selecting dependent measures for an evaluation, this is crucial to keep in mind. If the objective of an evaluation is related to one element, picking a success criterion related to another spells trouble. Imagine researching on-body interaction. ## 5. Discussion

We have outlined a user interface model composed of four elements and illustrated how that model helps us think about the concept of UI and how to design and evaluate UIs. Next, we answer some questions that the reader may have.

### 5.1. Is this model significant?

In the 1980s and 1990s, several papers dealt with the concept of UIs (e.g., Foley et al., 1984; Moran, 1981; Chignell, 1990). Since then,

general interest has faded. What is significant about discussing UIs again?

We argue that the proposed model offers three key advances over earlier work. First, we have shown that the user interface is not a boundary about which HCI has nothing to say. Rather, it has distinct elements and concerns. Second, we believe that it is essential to characterize user interface elements in a way that is not tied to a particular interaction style. Earlier work was applicable to just command-line interfaces, but the DIRA model helps to discuss the role and design concerns of devices in a way that works across, say, sensing systems (Bellotti et al., 2002) and implicit interaction (Serim and Jacucci, 2019). Third, we believe that separating tasks from operations, as well as the success of an interactive system from the success of an interaction technique, is an improvement over existing models.

We find that a dialog around the central concepts of HCI is currently lacking (as do Kostakos, 2015; Oulasvirta and Hornbæk, 2016), including the concept of user interface. An important and closely related concept is *interaction*. Hornbæk and Oulasvirta (2017) identified seven views of interaction. In our reading, many of them relate to our model of user interfaces. For instance, they discussed interaction-as-transmission, defined as “transmission of information between the computer and the user” (p. 5042). They explain how good interaction in this view is about achieving maximum throughput of information. Our model relates this to a particular element of UIs and their design concerns, namely interaction techniques. Thus, the UI is the justification for this concern and measure. Of course, there are plenty of other concerns and measures of goodness in HCI, but the fact that they may be derived from the model is a strength.

Other concepts include *principles* of interaction (e.g., Beaudouin-Lafon, 2000), *interaction styles*, and *qualities of interaction* (e.g., Hornbæk

et al., 2019). When discussing UIs in relation to these aspects of interaction, we currently think of styles or paradigms of user interfaces that are similar across one or more of the elements of the UI. A review of 35 years of the use of the concept of interaction at CHI (Hornbæk et al., 2019) summarized the frequent styles of interaction. We believe that some of them might be more easily expressed and understood in terms of our model. For instance, what the authors call modality is virtually always about the device and its sensing capabilities. What they call action is all about techniques. What they call characteristic, however, is about many aspects of the UI including device (e.g., physical, multimodal), assembly (e.g., spatial), and technique (e.g., direct). Thus, perhaps it is easier to discuss styles of interaction and characteristics of interaction when you are also specific about what UI element is being described.

We see no need to equate UIs and interaction, but the DIRA model provides a frame for discussing interaction and UIs together. It was surprising to us to see little consensus on what a UI is and many explanations which merely used synonyms of interaction or defined UIs exclusively through examples or users' tasks (see the Related Work section). We do not claim to have reached spotless clarity, but we do offer our attempt to update and refine a model of the UI as a way of debating a central concept in the field.

### 5.2. Is this model a step backwards?

In some views of HCI, it is important to expand the concept of UI or even move away from it altogether. For instance, Laurel and Mountford (1990) wrote that “When the concept of the interface first emerged, it was commonly understood as the hardware and software through which a human and a computer could communicate. As it has evolved, the concept has come to include the cognitive and emotional aspects of the user's experience as well.... The noun, interface is taken to be a discrete and tangible thing that we can map, draw, design, implement, and attach to an existing bundle of functionality. One of the goals [...] is to explode that notion and replace it with one that can guide our work in the right direction”. Similarly, Beaudouin-Lafon (2004) argued to focus less on user interfaces and more on interaction. Kuutti and Bannon (2014) encouraged a practice-oriented view of computing, highlighting the need to “focus on momentary and ahistorical HCI situations”. And (Benyon, 2019) moves from the UI towards people, activities, and contexts. So why go back to the concept of UI?

We think that a focus on the UI is important because it gives *analytic clarity*. For instance, the HCI field has followed the recommendation by Laurel and Mountford (1990) given above and included user experience in many of the work (i.e., experience design, Hassenzahl, 2010). We appreciate that work. But even research in that area has had to turn back to UI elements (like representations and interaction techniques) and understand their effects on experience. So while an expansion of concerns is important, the UI elements continue to be a major factor in shaping experience. In the model we have proposed, all of this is present, and all acknowledged as important. We also think that it is useful to separate some aspects of the user and some aspects of the UI. In several models, the users' tasks and the users' models of the system are written about as if they are part of the UI (e.g., Moran, 1981; Shneiderman et al., 2016). For instance, many models include tasks as part of the UI even though those are obviously something people do and obviously shaped by the available functionality. We find that unclear.

We believe that focusing on UIs is crucial also because it is strongly linked to *construction*. Although (Beaudouin-Lafon, 2004) speaks about interaction, many of his concepts are directly relevant to constructing user interfaces. For example, interaction techniques, as we understand them in the present model, are characterized by his concepts of indirection, integration, and compatibility (Beaudouin-Lafon, 2000). We still find the concept of interaction useful and important. As discussed, we find many views of interaction (from Hornbæk and Oulasvirta, 2017) directly related to the UI.

### 5.3. How may this model be used?

We see three main uses. First, we have shown how to use the definition to describe the elements of UIs. Perhaps such a structured approach has advantages in teaching. We have used the textbooks of Shneiderman and colleagues in teaching (e.g., Shneiderman et al., 2016) because they have descriptions of different interaction styles. Perhaps the proposed model could streamline and even clarify the presentation. Describing other emerging types of UIs, say, body-based UIs or brain-computer interfaces, may also be facilitated by the elements. Our examples of tangible user interfaces and mixed reality aimed to illustrate this.

Second, it is possible to think in new ways about the design and evaluation of UIs when their elements are made clear. Our examples of design and evaluative uses of the model intended to illustrate this.

Third, there are basic, and perhaps even philosophical, questions about when something is and is not a user interface. The model allows those questions to be answered by specifying the necessary and sufficient conditions for something to be considered a UI.

### 5.4. What remains to be done?

Many things! We have not made any attempt to cover how to work with the individual levels of the model. Earlier work contains plenty of diagramming techniques (e.g., state-transitions in Buxton, 1990) and notations (e.g., the User Action Notation, UAN in Hartson et al., 1990). Relating them to the model and its elements would be useful.

Our model is not intended to support the implementation or software architecture of UIs (those topics are dealt with for instance by Cockton and Gram, 1996; Basman et al., 2018), but could be positioned between the models of interaction (e.g., Hornbæk et al., 2019, as discussed above) and the UI development models (e.g., Meixner et al., 2011; Sanctorum and Signer, 2019; Dix and Runciman, 1985, as discussed in Related work). Establishing new links between the models of interaction (e.g., design concerns like throughput) and implementation (e.g., control like that of a cursor gain) through DIRA's elements (e.g., interaction technique) would be useful to both HCI research and UI development practices.

We acknowledge the need to more clearly articulate the implications of multiple users of UIs and of ecologies of devices (Bødker and Klok-mose, 2011). Some earlier models, such as PAC, entails that “compound objects can be built from elementary interactive objects” (Coutaz, 1987). In the case of DIRA, the elements work similarly to compound objects (i.e., they can be “nested” as we explain for instance in connection with Assemblies). Similarly, the objects of Arch model can be branched to multiple users and contexts (Beaudouin-Lafon, 1994). This could be the case of DIRA as well, but remains future work.

## 6. Conclusion

The user interface is central to HCI, but as a scientific concept, it remains fuzzy. Our goal in this paper has been to reinvigorate the discussion of the concept. We have described a model of four elements, their main roles, and central concerns. The model has been used to describe simple user interfaces, analyze user interfaces paradigms, and form questions for designing and evaluating user interfaces; it can also have other uses. Independently of the merits of the proposed model, we argue that returning to the user interface as a topic of discussion would be beneficial to the field of HCI.
